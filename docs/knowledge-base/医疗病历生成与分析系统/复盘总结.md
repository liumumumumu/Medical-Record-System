# 复盘总结

> [!note]
> 当前项目尚未结束。本页先记录 v0.2 前端阶段复盘，最终发布后再补完整数据和结论。

## v0.2 阶段复盘

### 做得好的地方

- 通过多轮区域化调整，明确了导航、版心和主任务区的实际尺寸，而不是一次性推翻重做。
- 将病例字段抽成配置，后续调整字段、必填和选项时不需要重写页面结构。
- 及时发现卡片嵌套问题，并把内部区域改为更符合任务型应用的分隔线布局。
- 在仅有首页时提前补上真实路由和结果页，避免导航长期停留在假状态。
- 补齐了错误、提交、空状态和刷新恢复，不只完成静态正常路径。
- 保持了“页面先完整、动画后统一”的节奏，避免动效掩盖结构问题。

### 不足

- 前期过度依赖截图和小幅像素调整，缺少提前统一的 CSS 令牌与布局规则。
- 最初把“模块独立”误解为每个模块都做成卡片，造成卡片套卡片。
- 当前仍没有真实后端、真实认证和自动化测试，功能完成度主要集中在前端演示层。
- 项目尚未建立正式 Git 提交，过程可追溯性不足。
- 当前未完成真实浏览器截图验收，代码尺寸正确不等于最终视觉完全正确。

## 技术沉淀

- 配置化表单适合字段频繁变化的课程项目，但配置必须同时包含类型、分组、必填、默认可见、范围和错误文案。
- 隐藏必填字段必须提供显式提示；提交失败时自动展开和聚焦是比静默禁用按钮更清晰的方案。
- 导航状态应由 React Router 驱动，而不是单独维护 `activeTab`。
- 临时结果可用 `sessionStorage` 保持会话连续，但真实数据应交给后端和权限体系。
- 任务型应用的模块独立主要依靠标题、间距和分隔线，不应默认使用卡片。

## AI 协作沉淀

- 用户偏好逐区域确认，AI 修改时应报告具体尺寸变化，避免“已经改了但视觉看不出”的沟通问题。
- 视觉审查应先区分营销页和任务型应用，避免把通用 SaaS 卡片模板套进医疗工作台。
- 每轮完成后应同时更新项目状态、任务中心、开发日志和 AI 接手上下文，避免下次从旧截图猜现状。

## 产品 / 业务 / 架构认知提升

- “结构化病历生成”和“医学诊断建议”是不同风险等级的能力，展示和文案必须区分。
- 医疗系统即使是课程演示，也要提前建立脱敏数据和免责声明边界。
- 前端结果页不能替代后端数据模型设计，下一阶段必须先锁定请求和响应协议。
- 固定高度不是最高目标，信息量、可读性和任务完成效率优先。
