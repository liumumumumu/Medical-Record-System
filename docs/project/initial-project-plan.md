# 医疗病历生成与分析系统项目计划

## Summary
本项目围绕“医疗病历生成与分析系统”展开，目标是实现一个贴合作业要求、可本地运行、可演示、可答辩的完整 Web 系统。系统支持患者信息采集、病历文本预处理、医学术语和症状识别、结构化病历生成、诊断辅助分析、治疗建议生成、历史记录管理和报告下载。

技术路线尽量遵循原始要求：采用 `Python + Java` 双后端思路，其中 `Java SpringBoot` 负责业务接口、用户管理、数据存储和系统整合，`Python Flask` 负责 NLP、病历生成和分析模型服务；前端采用 `React`，数据库采用 `MongoDB`，开发和配置管理使用 `Git`。AI/NLP 部分以 `Transformer / BioBERT 思路 + 规则词典 + 传统机器学习` 结合实现，保证既贴合作业要求，又能在小学期内落地。

## Key Changes
### 一、系统功能模块
- 数据采集：支持手动录入患者基本信息、主诉、现病史、既往史、体征、检查结果等字段；HIS/EMR 数据来源作为模拟导入或背景说明，不直接连接真实医院系统。
- 数据处理：完成文本清洗、去噪、去除无效字符、分词、停用词过滤、医学术语标准化和结构化字段转换。
- 病历生成：根据患者结构化信息和主诉文本，生成包含基本信息、主诉、现病史、既往史、体格检查、初步诊断、处理建议等内容的标准化病历。
- 病历分析：识别症状和体征，提取医学关键词，输出候选诊断、诊断依据和治疗建议。
- 用户界面：提供登录、患者信息录入、病历生成结果、分析结果、历史记录、报告下载等页面。
- 报告导出：支持将病历和分析结果导出为 `PDF` 或 `DOCX`。

### 二、技术栈设计
- 前端：`React + Vite + Ant Design + Axios`，负责页面展示、表单录入、结果可视化和接口调用。
- Java 后端：`SpringBoot`，负责登录认证、业务接口、MongoDB 数据存储、报告下载、调用 Python 分析服务。
- Python 服务：`Flask`，负责文本预处理、症状识别、病历生成、诊断分析和治疗建议生成。
- 数据库：`MongoDB`，存储用户信息、患者原始数据、生成病历、分析结果和报告记录。
- NLP/AI：`jieba / re / TF-IDF / scikit-learn / Transformer 或 BioBERT 预训练模型调用思路 / 医学词典规则`。
- 医疗标准：`SNOMED CT` 和 `HL7` 作为术语标准化、数据交换格式的设计参考；项目中实现简化版字段映射，不强制接入完整标准库。

### 三、四人分工
- A：AI 与病历分析负责人
  负责症状识别、医学术语抽取、诊断分类模型、BioBERT/Transformer 方案调研、治疗建议模板和 AI 模块说明文档。交付症状词典、诊断模型、分析接口说明和模型评估结果。

- B：数据处理与标准化负责人
  负责患者数据字段设计、样例数据构建、文本清洗、分词、停用词、医学术语标准化、训练集和测试集整理。交付数据字段规范、预处理脚本、样例数据集、医学词典和停用词表。

- C：后端与数据库负责人
  负责 SpringBoot 项目搭建、MongoDB 表/集合设计、用户登录、病历管理、历史记录、报告下载、调用 Python Flask 服务和统一接口返回。交付后端接口、数据库设计、API 文档和联调说明。

- D：前端与演示负责人
  负责 React 页面、路由、表单校验、病历展示、分析结果展示、编辑保存、下载按钮和最终演示流程。交付前端项目、页面原型、API 调用封装和演示脚本。

## Public APIs / Interfaces / Types
核心输入 `PatientInput`：
```json
{
  "name": "张三",
  "gender": "男",
  "age": 45,
  "chief_complaint": "发热咳嗽3天",
  "history_present_illness": "3天前受凉后出现发热、咳嗽、乏力",
  "past_history": "无高血压糖尿病史",
  "physical_exam": "体温38.5℃，咽部充血",
  "lab_results": "白细胞轻度升高"
}
```

核心输出 `AnalysisResult`：
```json
{
  "generated_record": "完整结构化病历文本",
  "symptoms": ["发热", "咳嗽", "乏力"],
  "medical_terms": ["上呼吸道感染", "白细胞"],
  "diagnosis_top1": "上呼吸道感染",
  "diagnosis_candidates": ["上呼吸道感染", "流感"],
  "diagnosis_reason": "患者存在发热、咳嗽、咽部充血等症状",
  "treatment_advice": "建议休息、多饮水，必要时完善血常规检查"
}
```

主要接口：
- `POST /api/auth/login`：用户登录
- `POST /api/patients`：提交患者信息
- `POST /api/records/generate`：生成病历
- `POST /api/analysis/run`：运行症状识别、诊断分析和治疗建议生成
- `GET /api/records`：查看历史记录
- `GET /api/records/{id}`：查看单条病历
- `PUT /api/records/{id}`：编辑保存病历
- `GET /api/reports/{id}/download`：下载报告

模块调用链：
`React 前端 -> SpringBoot 后端 -> Flask NLP/AI 服务 -> SpringBoot 存入 MongoDB -> React 展示结果`

## Test Plan
- 正常录入患者信息后，系统能生成完整结构化病历。
- 输入缺失必填字段时，前端和后端都能给出提示。
- 文本预处理能完成清洗、分词、停用词过滤和关键词提取。
- 症状识别能从样例病历中抽取发热、咳嗽、腹痛、乏力等常见症状。
- 诊断分析能输出至少 2-5 类常见疾病的候选诊断结果。
- 治疗建议能根据不同诊断标签生成不同建议。
- 病历可查看、编辑、保存并再次读取。
- 报告下载能导出完整病历和分析结果。
- 前端、SpringBoot、Flask、MongoDB 四部分能完成完整联调。
- 答辩时能展示数据采集、数据处理、病历生成、病历分析、报告导出完整流程。

## Assumptions
- 默认团队为 4 人，开发周期按 3 周左右安排。
- 不接入真实医院 HIS/EMR，只使用模拟数据或公开样例数据。
- 手写记录、图像、视频等非结构化数据作为扩展说明，不作为核心必做功能。
- BioBERT/Transformer 可作为预训练模型调用或方案展示，不进行大规模模型训练。
- SNOMED CT、HL7 作为标准化设计参考，项目实现简化字段映射即可。
- 治疗建议仅用于课程演示，页面和报告中需要注明“不替代医生诊断”。
