# 医疗病历生成与分析系统四人分工与交接说明

## 1. 文档目的

本文档用于明确“医疗病历生成与分析系统”四名成员的具体工作内容、阶段性交付物、模块边界和交接注意事项。项目整体采用：

`React 前端 -> SpringBoot 后端 -> Flask NLP/AI 服务 -> MongoDB 数据库`

系统目标是完成一个贴合作业要求、可本地运行、可演示、可答辩的医疗病历生成与分析系统。核心功能包括患者信息采集、数据预处理、病历生成、症状识别、诊断辅助分析、治疗建议生成、历史记录管理和报告下载。

## 2. 总体协作原则

1. 先统一字段和接口，再分别开发。
2. 所有人都以同一套 `PatientInput`、`AnalysisResult`、`RecordEntity` 为准。
3. A 和 B 先产出 AI/数据模块的可调用函数或接口，C 负责封装成系统 API，D 只对接 C 提供的后端接口。
4. 每个模块必须提供最少 2-3 条可运行测试样例，避免最后联调时才发现输入输出不一致。
5. 所有治疗建议和诊断结果都必须标注为“课程演示辅助结果，不替代医生诊断”。

## 3. A：AI 与病历分析负责人

### 3.1 工作定位

A 负责系统中最能体现“智能分析”和“医疗 NLP”的部分，包括症状识别、医学术语抽取、诊断辅助分析、病历生成策略和治疗建议生成。A 不负责页面、不负责数据库、不负责 SpringBoot 接口，只需要保证 AI 模块可以被 C 调用。

### 3.2 具体工作内容

1. 设计症状识别规则。
   - 建立常见症状词典，例如发热、咳嗽、咽痛、乏力、腹痛、腹泻、头痛、胸闷等。
   - 从主诉、现病史、体格检查、检查结果中识别症状。
   - 支持同义词或近义表达，例如“发烧”映射为“发热”，“拉肚子”映射为“腹泻”。

2. 设计医学术语抽取规则。
   - 建立医学术语词典，例如上呼吸道感染、流感、胃肠炎、高血压、糖尿病、白细胞、血常规等。
   - 输出标准化术语列表，供病历分析页面展示。
   - 在文档中说明 SNOMED CT 作为标准化参考，但项目内使用简化映射实现。

3. 实现诊断辅助分析。
   - 至少覆盖 2-5 类常见疾病，例如上呼吸道感染、流感、急性胃肠炎、咽炎、普通感冒。
   - 可以采用“规则打分 + 传统机器学习模型”结合的方式。
   - 如果训练数据不足，优先使用规则打分保证演示稳定。
   - 输出 `diagnosis_top1`、`diagnosis_candidates`、`diagnosis_reason`。

4. 实现病历生成。
   - 根据患者基本信息、主诉、现病史、既往史、体格检查、检查结果等字段生成结构化病历。
   - 病历内容应包括：基本信息、主诉、现病史、既往史、体格检查、辅助检查、初步诊断、处理建议。
   - 采用模板生成，不进行大规模生成式模型训练。

5. 实现治疗建议生成。
   - 根据诊断标签映射建议模板。
   - 输出内容应谨慎，避免绝对化诊疗表述。
   - 每条建议后建议加上“请结合医生进一步判断”或类似提示。

6. 整理 BioBERT / Transformer 方案说明。
   - 说明项目中如何参考 BioBERT 或 Transformer 思路处理医学文本。
   - 如果实际未加载大模型，需要说明原因：本地资源有限、课程 demo 优先保证可运行。
   - 可以将其作为“模型升级方案”写入答辩材料。

### 3.3 交付内容

A 需要交付以下文件或模块：

- `symptom_extractor.py`：症状识别模块。
- `medical_term_extractor.py`：医学术语抽取模块。
- `diagnosis_analyzer.py`：诊断辅助分析模块。
- `record_generator.py`：结构化病历生成模块。
- `treatment_rules.json`：诊断标签与治疗建议映射表。
- `diagnosis_labels.json`：支持的疾病类别和关键词说明。
- `model_report.md`：模型说明文档，包括输入、输出、方法、评价指标和限制。
- `ai_test_cases.json`：AI 模块测试样例。

### 3.4 输出接口要求

A 的核心输出必须符合 `AnalysisResult`：

```json
{
  "generated_record": "完整结构化病历文本",
  "symptoms": ["发热", "咳嗽", "乏力"],
  "medical_terms": ["上呼吸道感染", "白细胞"],
  "diagnosis_top1": "上呼吸道感染",
  "diagnosis_candidates": ["上呼吸道感染", "流感"],
  "diagnosis_reason": "患者存在发热、咳嗽、咽部充血等表现，符合上呼吸道感染特征",
  "treatment_advice": "建议休息、多饮水，必要时完善血常规检查。本结果仅用于课程演示，不替代医生诊断。"
}
```

### 3.5 交接注意事项

- A 必须提前告诉 B 需要哪些输入字段，例如 `chief_complaint`、`history_present_illness`、`physical_exam`、`lab_results`。
- A 必须告诉 C 每个函数的调用方式、参数格式和返回格式。
- A 的模块不能直接读写 MongoDB，所有落库由 C 完成。
- A 的输出字段名不能随意改，改动必须同步通知 C 和 D。
- A 要准备固定 demo 样例，确保答辩时结果稳定可复现。

## 4. B：数据处理与标准化负责人

### 4.1 工作定位

B 负责系统的数据基础，包括患者字段规范、样例数据、文本清洗、分词、停用词、医学词典和训练测试数据。B 的结果是 A 进行模型分析、C 进行接口校验、D 设计表单字段的共同基础。

### 4.2 具体工作内容

1. 设计患者输入字段。
   - 确定前端录入和后端接收的统一字段。
   - 字段至少包括姓名、性别、年龄、主诉、现病史、既往史、体格检查、检查结果。
   - 明确哪些字段必填，哪些字段选填。

2. 构建样例数据。
   - 至少准备 10 条患者样例数据。
   - 至少覆盖 2-5 类疾病。
   - 每条样例应包含症状关键词和对应诊断标签。

3. 文本清洗与预处理。
   - 去除无效符号、HTML 标签、异常空格和重复标点。
   - 统一大小写、数字、单位和常见医学表达。
   - 对中文文本进行分词和停用词过滤。

4. 建立词典资源。
   - 症状词典：发热、咳嗽、腹痛等。
   - 医学术语词典：疾病、检查项目、体征、常见药物或处理方式。
   - 停用词表：的、了、和、以及等无分析意义词语。
   - 同义词表：发烧 -> 发热，拉肚子 -> 腹泻。

5. 准备模型训练和测试数据。
   - 输出 `train.csv`、`test.csv`、`demo_data.csv`。
   - 每条数据应包含文本字段和诊断标签。
   - 如果数据量不足，可以以规则样例为主，模型训练作为辅助说明。

6. 输出标准化 JSON。
   - 将用户输入转换为统一结构。
   - 保证 A 和 C 拿到的数据字段完全一致。

### 4.3 交付内容

B 需要交付以下文件：

- `data_schema.md`：患者输入字段规范。
- `preprocess.py`：文本清洗和标准化脚本。
- `feature_builder.py`：TF-IDF 或关键词特征构建脚本。
- `symptom_dict.txt`：症状词典。
- `medical_terms.txt`：医学术语词典。
- `stopwords.txt`：停用词表。
- `synonyms.json`：同义词映射表。
- `train.csv`：训练数据。
- `test.csv`：测试数据。
- `demo_data.csv`：演示数据。

### 4.4 输出接口要求

B 输出的标准化数据建议为：

```json
{
  "name": "张三",
  "gender": "男",
  "age": 45,
  "chief_complaint": "发热咳嗽3天",
  "history_present_illness": "3天前受凉后出现发热、咳嗽、乏力",
  "past_history": "无高血压糖尿病史",
  "physical_exam": "体温38.5℃，咽部充血",
  "lab_results": "白细胞轻度升高",
  "clean_text": "发热 咳嗽 乏力 咽部充血 白细胞升高",
  "tokens": ["发热", "咳嗽", "乏力", "咽部充血", "白细胞"]
}
```

### 4.5 交接注意事项

- B 必须先把 `data_schema.md` 给 A、C、D，作为所有人开发依据。
- B 的词典文件要保证编码为 `UTF-8`，避免中文乱码。
- B 改字段名时必须通知 C 和 D，否则接口会无法联调。
- B 给 A 的训练数据必须带诊断标签，否则 A 无法训练或评估模型。
- B 给 D 的字段说明要写清楚必填项、选填项和错误提示规则。

## 5. C：后端与数据库负责人

### 5.1 工作定位

C 负责 SpringBoot 后端、MongoDB 数据存储、统一 API、登录认证、病历管理、报告下载和 Python Flask 服务调用。C 是系统整合核心，负责把 A、B、D 的模块串起来。

### 5.2 具体工作内容

1. 搭建 SpringBoot 项目。
   - 配置基础项目结构。
   - 配置 MongoDB 连接。
   - 配置跨域，保证 React 前端可以访问接口。
   - 配置统一返回格式。

2. 实现用户登录。
   - 支持简单账号密码登录。
   - 返回 token 或模拟 token。
   - 不做复杂权限系统。

3. 设计 MongoDB 集合。
   - `users`：用户信息。
   - `patient_records`：患者输入、生成病历、编辑后病历。
   - `analysis_results`：症状识别、诊断结果、治疗建议。
   - `report_files`：报告文件信息。

4. 实现核心业务接口。
   - 接收前端患者信息。
   - 调用 Flask NLP/AI 服务生成病历和分析结果。
   - 将结果保存到 MongoDB。
   - 支持查询历史记录和单条记录。
   - 支持编辑保存病历。
   - 支持下载 PDF 或 DOCX 报告。

5. 调用 Python Flask 服务。
   - 使用 HTTP 请求调用 Flask 的 `/nlp/analyze` 或类似接口。
   - 处理 Flask 服务不可用、返回异常、字段缺失等情况。
   - 保证对前端返回统一格式。

6. 编写 API 文档。
   - 说明每个接口的 URL、请求方法、请求 JSON、返回 JSON。
   - 给 D 提供可直接调用的接口示例。

### 5.3 交付内容

C 需要交付以下内容：

- SpringBoot 后端项目。
- MongoDB 集合设计文档。
- `application.yml` 或 `.env.example` 配置示例。
- 登录、患者提交、病历生成、分析、历史记录、编辑、下载接口。
- `api_document.md`：接口文档。
- 报告导出功能。
- 后端启动说明和联调说明。

### 5.4 统一返回格式

所有接口建议统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

错误返回示例：

```json
{
  "code": 400,
  "message": "缺少必填字段 chief_complaint",
  "data": null
}
```

### 5.5 主要接口

- `POST /api/auth/login`：用户登录。
- `POST /api/patients`：保存患者信息。
- `POST /api/records/generate`：生成结构化病历。
- `POST /api/analysis/run`：执行症状识别、诊断分析和治疗建议生成。
- `GET /api/records`：查询历史记录。
- `GET /api/records/{id}`：查询单条病历。
- `PUT /api/records/{id}`：编辑保存病历。
- `GET /api/reports/{id}/download`：下载报告。

### 5.6 交接注意事项

- C 必须等 B 的 `data_schema.md` 确定后再最终确定接口字段。
- C 调用 A 的 Flask 服务时，必须保留原始输入和分析结果，方便答辩展示。
- C 给 D 的 API 文档必须包含真实可运行示例。
- C 不能让前端直接访问 MongoDB 或 Flask AI 服务，D 只访问 SpringBoot。
- C 要提前准备接口异常处理，避免 Flask 没启动时整个系统崩溃。
- C 的下载接口需要和 D 约定：是直接返回文件流，还是返回下载地址。

## 6. D：前端与演示负责人

### 6.1 工作定位

D 负责 React 前端页面、用户交互、接口调用、结果展示和最终演示体验。D 不直接操作数据库，不直接调用 Flask 服务，不改 AI 逻辑，只对接 C 提供的 SpringBoot API。

### 6.2 具体工作内容

1. 搭建 React 项目。
   - 使用 `React + Vite + Ant Design + Axios`。
   - 配置页面路由。
   - 封装 API 请求方法。

2. 实现登录页。
   - 用户输入账号密码。
   - 调用 `/api/auth/login`。
   - 登录成功后保存 token 或登录状态。

3. 实现患者信息录入页。
   - 表单字段包括姓名、性别、年龄、主诉、现病史、既往史、体格检查、检查结果。
   - 必填字段校验。
   - 提交后调用后端接口。

4. 实现病历生成结果页。
   - 展示生成的结构化病历。
   - 支持局部编辑。
   - 支持保存编辑结果。
   - 支持跳转查看分析结果。

5. 实现分析结果页。
   - 展示症状列表。
   - 展示医学术语。
   - 展示预测诊断和候选诊断。
   - 展示诊断依据和治疗建议。
   - 明确展示“不替代医生诊断”的提示。

6. 实现历史记录页。
   - 展示已有患者记录。
   - 支持点击查看详情。
   - 支持下载报告。

7. 准备最终演示流程。
   - 准备固定演示账号。
   - 准备 2-3 条稳定 demo 数据。
   - 写出演示脚本：登录 -> 录入 -> 生成病历 -> 查看分析 -> 编辑保存 -> 下载报告。

### 6.3 交付内容

D 需要交付以下内容：

- React 前端项目。
- 页面路由配置。
- 登录页、患者录入页、病历结果页、分析结果页、历史记录页。
- API 调用封装文件。
- 表单校验规则。
- 演示数据和演示脚本。
- 前端启动说明。

### 6.4 页面字段要求

患者录入页至少包含：

- 姓名：必填。
- 性别：必填。
- 年龄：必填，限制为合理数字。
- 主诉：必填。
- 现病史：必填。
- 既往史：选填。
- 体格检查：选填。
- 检查结果：选填。

分析结果页至少展示：

- 生成病历。
- 症状识别结果。
- 医学术语提取结果。
- 初步诊断。
- 候选诊断。
- 诊断依据。
- 治疗建议。
- 免责声明。

### 6.5 交接注意事项

- D 必须严格按照 C 的 API 文档调用接口，不直接拼接不确定字段。
- D 的表单字段名要和 B 的 `data_schema.md` 保持一致。
- D 如果发现接口返回字段不够用，先和 C 沟通，不直接绕过后端。
- D 需要准备“接口失败”的页面提示，例如后端未启动、分析失败、下载失败。
- D 最终演示时要使用固定 demo 数据，避免现场临时输入导致结果不可控。

## 8. 建议开发时间安排

### 第 1 阶段：接口和数据先定

- C 搭建 SpringBoot 和 MongoDB 基础框架。
- D 搭建 React 页面静态原型。

- B 完成预处理脚本和 demo 数据。

- C 完成患者提交、病历生成、分析结果接口。

## 11. 风险和解决方案

1. 风险：BioBERT 或 Transformer 跑不动。
   解决：保留方案说明，实际使用规则词典和传统模型实现稳定 demo。

2. 风险：训练数据不足。
   解决：优先采用规则打分和关键词匹配，训练模型作为辅助展示。

3. 风险：Java 调 Python 联调失败。
   解决：先用 Postman 测 Flask，再让 SpringBoot 调用；保留本地模拟返回作为兜底。

4. 风险：MongoDB 环境安装困难。
   解决：使用本地 MongoDB 或 Docker MongoDB；如确实不可用，临时用 JSON 文件模拟，但答辩中仍说明设计数据库为 MongoDB。

5. 风险：前后端字段不一致。
   解决：以 B 的 `data_schema.md` 和 C 的 `api_document.md` 为唯一标准，任何字段改动必须全组同步。

6. 风险：治疗建议表述过于绝对。
   解决：统一使用“建议”“可考虑”“必要时”等表述，并加免责声明。

## 12. 最终验收标准

项目完成时应满足：

- 能本地启动 React 前端、SpringBoot 后端、Flask AI 服务和 MongoDB。
- 能完成登录、患者信息录入、病历生成、病历分析、编辑保存、历史记录和报告下载。
- 至少有 10 条 demo 患者数据。
- 至少支持 2-5 类常见疾病分析。
- 每个人都有明确可展示的成果。
- 答辩时能说明数据采集、数据处理、NLP 方法、病历生成、诊断分析、系统架构和分工协作。
- 页面和报告中明确说明系统结果仅用于课程演示，不替代医生诊断。
